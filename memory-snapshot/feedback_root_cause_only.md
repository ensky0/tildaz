tildaz 저장소 작업 시 모든 버그는 **진짜 원인을 찾아 그 자리에서 고침**.
증상만 가리는 fix 는 절대 채택 안 함:

- "dummy 첫 instance" 같은 GPU quirk 우회
- "+1 / -1 보정" 같은 off-by-one 추정 fix
- "이렇게 두면 일단 보이긴 한다" 류의 hack
- 다른 코드의 의미를 흐리는 우회

**Why:** 사용자가 명시적으로 강조 (2026-04-30). hack 은 (1) 다른 곳에서 더 큰
버그로 나타나고 (원인이 남아 있으니), (2) 다음 사람이 의미 못 알아보고,
(3) 미해결 quirk 는 코드 안 hack 보다 코드 옆 주석 + follow-up 이슈가 더 나음
("여기서 막혔고 원인 모름" 을 정직하게 남기는 게 후속 디버깅에 도움). 같은
규칙이 `AGENTS.md` 의 "근본 해결 원칙" 섹션에도 명시됨.

**How to apply:** tildaz 코드를 고칠 때마다.

- 한 번에 여러 hypothesis 를 hack 으로 시도하지 않고, 진단으로 사실을 좁힘
  → 진짜 원인을 1 발에 끝냄.
- 원인을 못 찾을 땐 "이 부분 원인 모름, 진단 더 필요" 라고 솔직히 보고.
- "잠시 hack 으로 두고 나중에 고치기" 도 안 됨 — 영구 hack 이 됨.
