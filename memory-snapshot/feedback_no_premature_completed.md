빌드 통과 + 한 번 띄워서 죽지 않는 것 (smoke launch) 만으로 "완료" 라고 표현하면 안 돼요. 특히 사용자 시각 / 입력 검증이 필요한 항목 (멀티탭, 키바인딩, 탭바 렌더, IME, 마우스, 종료 흐름 등) 은 사용자가 직접 손으로 시연해서 OK 한 후에만 commit message / 이슈 댓글에 "완료" 사용.

**Why:** 사용자가 commit log / 이슈 댓글을 신뢰해야 작업 추적이 가능해요. "완료" 가 거짓이면 다음 작업 결정 시 잘못된 전제로 진행하게 되고, 미해결 quirk 가 다른 milestone 진행 중에 뒤늦게 터져 비용이 큼. 또 사용자가 검증을 안 했는데 완료라고 보고받으면 본인이 한 게 사라진 듯한 좌절.

**How to apply:**
- commit message 의 검증 줄: "빌드 + smoke launch 통과. 사용자 환경 검증 미완 — 다음 항목 검증 필요: ..." 같이 *상태 그대로* 적기.
- 이슈 댓글 제목: "M11.x 완료" 가 아니라 "M11.x 코드 변경 + 빌드 통과 — 사용자 검증 대기" 같이 사실 그대로.
- "완료" 라는 단어는 사용자가 명시적으로 "잘 된다 / OK 야" 라고 확인 후에만 사용.
- 다음 작업으로 넘어갈 때 사용자 확인을 한 번 받고 진행.
- **commit / amend / force-push 자체도 사용자 시연 OK 후에만**. 빌드 + smoke launch 통과는 commit 의 *전제 조건* 이지 *사유* 가 아님. 사용자가 직접 띄워서 동작 확인하기 전엔 코드 변경 후 시연 요청만 하고 commit 보류. 사용자 OK 받으면 그때 amend (의도 같으면) 또는 새 commit.
- 예외: 분명히 mechanical 한 변경 (typo, 주석, doc-only) 이고 사용자가 명시적으로 "그냥 commit 해" 한 경우.
